# JavaScript が動く仕組み

どんぐり「じゃあさ、JavaScript ってどういう仕組みで動いているの？」

りすりす「JavaScript の動く仕組みを理解するためにはまずコンピュータの仕組みを知ってないけど」  
りすりす「コンピュータがどういう仕組みで動いているか知ってる？」

どんぐり「うーん、回路で動いてるってことは分かるよ！」

りすりす「それはまあそうなんだけど、一から教えるね」

## コンピュータの仕組み

りすりす「コンピュータの中身は大体こうなってるんだけど」

![コンピュータの中身](https://kuina.ch/images/l6prog/basic1/disassembling.png)
*コンピュータの中身 (&copy;Kuina-chan)*

りすりす「**『CPU』ってのは、プログラムを動かすコンピュータの脳みそとも言われる部品** なんだ」  
りすりす「**『RAM』(メモリ) は、一時的にデータを置いておく部品** だよ」

どんぐり「コンピュータにとっての作業机って感じかな」

りすりす「**『HDD』(ハードディスク) や『SSD』は、大量のデータをしまっておく部品** だね」

どんぐり「これは倉庫みたいな感じか」

りすりす「で、**これらを繋いでいるのが『マザーボード』っていうプリント基板** なんだ」

どんぐり「なるほど」  
どんぐり「なんとなくコンピュータの仕組みがわかったんだけど」  
どんぐり「つまり JavaScript は CPU が動かしてるってこと？」

りすりす「最終的にはそうなるんだけど、実は CPU には JavaScript を動かす機能なんてないんだ」

どんぐり「なんやて！？」

りすりす「**CPU が動かせるのは『マシン語』(機械語/ネイティブコード) っていう言語だけ** なんだ」

どんぐり「それはどんな言語なの？」

りすりす「ソースコードが文字列じゃなくて、データそのままって感じかな」  
りすりす「昔は 2 進数をスイッチで直接打ち込んでたらしいよ」

どんぐり「大変そう......」

:::details なぜ 2 進数？
どんぐり「でもなんで 2 進数なの？」

りすりす「**コンピュータは電圧が低い状態を `0`、高い状態を `1` として表現していて**」  
りすりす「**2 進数はコンピュータにとって都合がいい** んだ」  
りすりす「2 進数のほうが回路もシンプルにできるしね」  
:::

りすりす「そこで生み出されたのが **アセンブリ言語 (assembly language)** っていうプログラミング言語なんだ」

どんぐり「それはどういう言語なの？」

りすりす「マシン語を 16 進数で表してから、それぞれの数字を分かりやすい単語に一対一で対応付けられた言語だね」  
りすりす「アセンブラ (assembler) っていうプログラムでアセンブリ言語をマシン語に変換して動かすんだ」

どんぐり「どんなコードなんだろ......」

りすりす「たとえばアセンブリで `Hello, World!` を出力するプログラムは」

```asm
section .data
    msg db "Hello, World!", 0ah

section .text
    global _start
_start:
    mov rax, 1
    mov rdi, 1
    mov rsi, msg
    mov rdx, 13
    syscall
    mov rax, 60
    mov rdi, 0
    syscall
```

りすりす「こう書くよ」

:::message
りすりす「CPU のメーカー (Intel や AMD など) によってマシン語は微妙に違って」  
りすりす「マシン語が違うとアセンブリ言語も違うから」  
りすりす「これはあくまで一例だよ」  
:::

どんぐり「`Hello, World!` を出力するだけなのにすごい複雑そう......」

りすりす「うん、アセンブリ言語でも大変だったんだ」  
りすりす「でも、**FORTRAN** というプログラミング言語が出てきてからはプログラミングのハードルは大きく下がったらしいよ」  
りすろす「そして、様々な言語ができたんだ」

## プログラミング言語の種類

どんぐり「アセンブリ以外のプログラミング言語ってどういう仕組みで動くの？」

りすりす「**プログラミング言語は動く仕組みは大きく 2 種類に分けられる** よ」  
りすりす「一つは『**コンパイラ型言語**』(compiler language) で、もう一つは『**インタプリタ型言語**』(interpreter language is) って言うんだ」

### コンパイラ型言語

りすりす「まずコンパイラ型言語っていうのは」  
りすりす「**ソースコードをマシン語などに変換してから動かす言語** だよ」

![コンパイル](https://kuina.ch/images/l6prog/basic2/compilation.png)
*コンパイル (&copy;Kuina-chan)*

りすりす「**コンパイラ型言語をマシン語などに変換することを『翻訳』とか『コンパイル』って言って**」  
りすりす「**コンパイルをするプログラムのことを『コンパイラ』って言う** んだ」

:::message
りすりす「実は、コンパイラ型言語のコンパイル先はマシン語とは限らないんだ」  
りすりす「定義にもよるんだけど、インタプリタ型言語にコンパイルするっていう言語もあるよ」  
:::

どんぐり「FORTRAN はコンパイラ型言語なの？」

りすりす「うん」  
りすりす「他にも C 言語とその派生の C++、あとは Go とか Rust とかもコンパイラ型だね」

### インタプリタ型言語

:::details JavaScript はコンパイラ型言語でもある！？

JavaScript は手動でコンパイルをせずにそのまま実行できるので、一見するとインタプリタ型言語のように思えます。  
JavaScript は「V8」や「SpiderMonkey」などのプログラムによって動きます。

JIT コンパイラとは Just-In-Time コンパイラの略で、プログラムの実行時にコンパイルをしてから実行するというものです。  
ここでは説明するのが難しいので、インタプリタ型言語とコンパイラ型言語の中間、と理解しといてください。  

![TODO: イメージ画像]()
*JavaScript の JIT コンパイラ*

ブラウザや Node.js はそれぞれ JavaScript エンジンを内蔵しているため、  
JavaScript を動かすことができるのです。  
例えば、 Google Chrome ブラウザや Node.js には V8 エンジンが使われていて、  
Firefox ブラウザには SpiderMonkey が使われています。  

ちなみに Java や C# などの言語も JIT コンパイラを採用していますが、  
JavaScript と違ってこれらは手動で中間言語にコンパイルしとく必要があります。  
ただしその分実行時にコンパイルをしないためプログラムをすぐに動かすことができます。  
またコンパイラ言語とも違い、コンパイル後のファイルがマシン語ではないため CPU の種類が違くても動かすことができます。  
:::
